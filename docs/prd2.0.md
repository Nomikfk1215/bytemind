# ByteMind PRD

## 1. 文档信息

- 产品名称：ByteMind
- 文档版本：PRD
- 产品形态：本地 AI Coding CLI
- 目标平台：macOS / Linux / Windows
- 技术实现：Go
- 文档类型：产品需求文档

## 2. 产品定义

ByteMind 是一款面向本地代码仓库的 AI Coding CLI，帮助开发者在一个可持续会话的命令行环境中完成“理解项目、修改代码、执行命令、验证结果、继续任务、沉淀技能”的工程闭环。

ByteMind 以 CLI 作为主交互形态，以 Slash 命令和非交互式 Shell 调用作为补充入口。产品目标不是替代 IDE，也不是做成沉重的平台，而是在开发者最常用的 Terminal 场景里，提供一套足够强、足够稳、足够可控、且能沉淀重复流程的工程执行助手。

一句话定义：

> ByteMind 是一个运行在本地工作区中的 AI Coding CLI，让开发者在 Terminal 内完成“理解—修改—执行—验证—恢复—沉淀”的工程闭环。

## 3. 立项背景与产品目标

### 3.1 立项背景

开发者对 AI 编程工具的需求，正在从“回答问题”转向“直接执行任务”。仅提供问答、解释、补全，已经无法满足真实开发流程中的连续工作需求。用户真正需要的是：

- 能直接读取仓库上下文，而不是只根据几段粘贴代码回答
- 能修改文件、运行命令、检查结果，而不是只给建议
- 能保持任务连续性，在多轮交互中持续推进同一工程问题
- 能在具备执行能力的同时，保持边界清晰、过程透明、风险可控

CLI 仍然是当前阶段最适合的产品形态，原因如下：

1. 最符合目标用户习惯  
核心用户本身就在 Terminal 内工作，终端是最低切换成本的入口。

2. 与现有工作流结合  
可直接运行于本地仓库、shell 脚本等常见终端工作流中。

3. 更利于脚本化与自动化  
Slash 命令和非交互入口可以自然兼容 shell 脚本和 CI 场景。

4. 实现复杂度仍可控  
以 Go 实现 CLI，可以最大化复用 runtime、tool 调度、会话管理与配置体系，避免额外维护复杂 UI 状态。

### 3.2 产品目标

1. 让用户能够在本地工作区中通过自然语言完成真实的软件工程操作。
2. 让用户在命令行中持续推进一个任务，而不是只能进行一次性问答。
3. 让系统具备项目理解、代码修改、命令执行和结果验证能力，形成完整闭环。
4. 让执行过程足够清晰、可审查、可恢复，提升用户对 agent 的信任。
5. 用户可接入自定义 skills，并且高频工程流程能够以用户自定义 skills 的形式被沉淀与复用。
6. 在 Go 技术栈下保持可持续迭代，严格控制范围，避免产品形态膨胀。

### 3.3 产品原则

1. 本地优先：默认围绕本地工作区运行，不依赖云端项目索引。
2. CLI 优先：CLI 是主交互入口，Slash 命令与非交互调用是补充入口。
3. 执行优先：重点不是聊天体验，而是完成任务。
4. 可控优先：文件修改、命令执行、越界操作必须具备边界与确认机制。
5. 恢复优先：中断、失败、上下文长度溢出的结果。
6. 可沉淀优先：高价值重复流程应能被整理为用户自定义 skills，而不是每次从头描述。

### 3.4 成功标准

当用户输入以下类型任务时，ByteMind 应能稳定完成主要流程：

- “分析这个仓库的结构并给我改造建议。”
- “在当前模块里加一个配置项，并更新读取逻辑。”
- “定位登录测试失败原因并修复。”
- “继续完成上一次未做完的修改。”

闭环成立的标准是：

- 系统能读取本地上下文
- 系统能调用工具执行实际操作
- 系统能输出明确结果
- 用户知道改了什么、执行了什么、下一步怎么继续

## 4. 用户画像 & 场景

### 4.1 目标用户

ByteMind 的目标用户主要包括三类：

- 终端优先的后端开发者
- 多语言的全栈/工具型开发者
- 小团队技术负责人或核心维护者

### 4.2 用户画像

#### 4.2.1 小李：终端优先的后端开发工程师

基本信息：

- 29 岁，后端开发，主要负责 Go 服务和内部 API
- 日常在 Terminal、SSH、VS Code 之间切换
- 熟悉 Git、测试命令、构建命令，希望尽量在终端里闭环完成工作

工作习惯：

- 进入一个仓库后，先看目录结构、配置入口和服务启动方式
- 修改代码前通常会先全局搜索关键词，再读关键文件
- 修改后会立刻跑测试、构建或脚本验证结果
- 遇到中断时，希望第二天直接接着昨天的进度继续做

核心目标：

- 快速理解陌生仓库和陌生模块
- 让 Agent 帮他做真实改动，而不是只给建议
- 在终端里完成“分析 -> 修改 -> 验证 -> 总结”

主要痛点：

- 通用聊天式 AI 需要反复喂上下文，理解仓库成本高
- 改动以后看不清具体改了哪些文件、跑了哪些验证
- 复杂任务被打断后，恢复成本高

#### 4.2.2 小王：多语言的全栈/工具型开发者

基本信息：

- 27 岁，全栈开发，既写前端，也写后端、脚本和配置
- 经常需要在一个仓库中同时修改多个文件，或临时新增脚本与模板
- 喜欢把常用命令封装成 shell 脚本或自动化流程

工作习惯：

- 接到任务后，会先让工具帮他确认现有项目结构和规范
- 对重复性工作非常敏感，倾向于让 Agent 直接生成可落地结果
- 既会交互式使用工具，也会在非交互模式下批量触发任务
- 关注结果是否清晰、输出是否适合复制到 issue、PR 或群里

核心目标：

- 在现有项目结构下快速新增文件、脚本、配置或模板代码
- 让一次任务尽量少切换上下文，最好在一次会话里推进完成
- 让终端 Agent 真正融入自己的 shell 工作流
- 把重复出现的工程流程沉淀为可复用的用户自定义 skills

主要痛点：

- 很多工具只能“说怎么做”，不能“替他做完”
- 多文件任务容易丢上下文，结果不稳定
- 参数体系不统一时，工具学习和切换成本高

#### 4.2.3 小黄：小团队技术负责人兼核心维护者

基本信息：

- 33 岁，技术负责人，既做关键代码评审，也会亲自处理复杂改造
- 关注研发效率，也关注风险、治理和团队采纳成本
- 希望先在低风险、高频任务上引入 Coding Agent

工作习惯：

- 评估新工具时，会先看边界是否清晰、输出是否可解释、失败是否可恢复
- 希望团队成员在同一套规范下使用工具，而不是各自摸索
- 特别关心危险命令、越权改动和不可追踪的文件修改
- 对“继续上次任务”能力敏感，因为很多任务本身就是多天推进

核心目标：

- 让团队在本地仓库内安全地使用 Agent
- 降低重复分析、重复修复和重复验证的时间成本
- 让每次执行都有记录、有摘要、可回溯

主要痛点：

- 很多 Agent 看起来能力很强，但实际难治理、难解释、难落地
- 缺少清晰的审批、阻断和变更追踪机制
- 团队任务一旦中断，就很难恢复到准确的上下文状态

### 4.3 用户痛点

三类用户的共性痛点可归纳为：

- 项目理解成本高，AI 需要反复补上下文
- 很多 AI 工具只能建议，不能真实落盘和执行
- 修改后的结果不可追踪，用户看不清实际变更
- 测试、构建和修复过程不连贯，闭环常被打断
- 中断后难恢复，任务容易重新分析一遍
- 用户既希望 Agent 能执行，又担心它乱执行
- 高频任务流程难以复用，每次都要重新描述和重复验证

### 4.4 用户故事

#### 4.4.1 小李的用户故事

- 作为小李，我进入一个陌生 Go 仓库时，希望先用 ByteMind 快速分析项目结构、配置入口、关键模块和调用链，这样我不用手动粘贴上下文就能开始工作。
- 作为小李，当我确认修改方向后，希望 ByteMind 能直接在仓库里改代码、补参数校验、修函数逻辑，并在改完后立刻运行测试或构建，这样我可以在终端里完成完整闭环。
- 作为小李，当任务做到一半被打断时，我希望第二天重新进入仓库后，ByteMind 能告诉我上次改了哪些文件、跑了哪些命令、卡在什么地方，并允许我继续执行，这样我不用从头再来。

#### 4.4.2 小王的用户故事

- 作为小王，当我需要新增脚本、配置、模板文件或一个小模块时，我希望 ByteMind 能先理解当前项目结构和约定，再直接生成并落盘结果，而不是只返回一段建议代码。
- 作为小王，当我处理一个涉及多个文件的任务时，我希望 ByteMind 能在一次会话中持续推进“先查找、再修改、再验证、最后总结”的过程，这样我能减少上下文切换和重复描述。
- 作为小王，当我把 ByteMind 接入 shell 脚本或 CI 时，我希望它支持非交互模式，且输出结构化、可复制，这样我可以把它纳入现有终端工作流。
- 作为小王，当我反复执行某类相似任务时，我希望能把这套做法保存为本地 skill，并在后续仓库中直接复用，这样我不用每次都重新描述流程。

#### 4.4.3 小黄的用户故事

- 作为小黄，当我评估 ByteMind 是否适合团队落地时，我希望它默认限制在当前工作区内操作，Shell 命令具备审批和阻断机制，高风险动作必须确认，这样我才敢让团队真实使用。
- 作为小黄，当团队成员使用 ByteMind 修改代码时，我希望它能明确告诉我改了哪些文件、执行了哪些命令、验证是否通过，这样我可以快速 review 结果，而不是面对黑盒输出。
- 作为小黄，当一个复杂任务需要跨多天推进时，我希望 ByteMind 能恢复上次会话主题、任务进度和阻塞原因，并给出继续建议，这样团队不会因为上下文丢失而反复做同样的分析工作。

### 4.5 用户旅程

#### 阶段一：进入仓库

- 用户进入目标仓库目录
- 启动 `bytemind` 或 `bytemind run`
- 希望系统能够识别当前工作区并快速可用

#### 阶段二：理解任务与仓库

- 用户输入自然语言任务
- 系统浏览目录、读取文件、搜索关键词
- 用户确认系统已经理解项目结构和目标范围

#### 阶段三：执行修改

- 系统进行定点修改、新建文件或 patch 更新
- 用户希望知道系统改了什么、为什么这么改

#### 阶段四：执行验证

- 系统运行测试、构建或脚本命令
- 根据结果继续修复，或给出明确的完成状态

#### 阶段五：总结与恢复

- 系统输出变更摘要、命令摘要和结果总结
- 若任务中断，用户下次进入时可恢复会话并继续

### 4.6 核心场景

- 仓库理解：分析项目结构、定位入口、理解调用链
- 定向代码修改：增加配置项、修复逻辑、补充校验
- 新建文件或模块：生成脚本、模板、配置和小模块
- 命令执行与验证：运行测试、构建、脚本并基于结果修复
- 连续迭代：在一次会话中完成分析、修改、验证、修复
- 恢复未完成任务：继续上次会话主题、进度与阻塞点
- 技能沉淀：把常见修复、初始化和验证流程保存为skills

### 5.3 功能描述

#### 5.3.1 交互式 CLI 会话

- 用户可进入持续会话，连续输入需求
- 系统保留上下文和执行状态
- 会话过程中应持续输出任务状态、工具结果和结果摘要
- 适合复杂任务的多轮推进

#### 5.3.2 单次任务模式

- 用户通过单条命令直接触发任务
- 适合快速场景、shell 和自动化调用
- 执行完成后输出结果摘要并退出

#### 5.3.3 项目理解能力

- 系统可浏览目录结构
- 读取文本文件和关键配置
- 搜索关键词、符号和调用关系
- 结合仓库实际内容解释代码逻辑

#### 5.3.4 文件修改能力

- 支持精确修改已有函数和逻辑
- 支持 patch 形式修改多个位置
- 支持创建新文件、新脚本和新模块
- 修改后输出变更摘要

#### 5.3.5 命令执行与验证能力

- 支持执行测试、构建和脚本命令
- 读取标准输出、错误输出和退出码
- 根据执行结果继续修复或给出结论

#### 5.3.6 会话恢复能力

- 支持保存历史会话和最近一次任务状态
- 支持恢复上次会话主题、已执行命令、已修改文件和阻塞原因
- 支持在恢复后继续推进任务

#### 5.3.7 用户自定义 skills

- 支持用户在本地工作区或个人配置目录定义 skills 文件
- skills 可描述适用场景、执行提示、推荐验证步骤和输出要求
- 用户可在 CLI 内查看可用 skills，并按名称触发执行
- 本阶段不做远程 skills 市场，只做用户可定义、可复用的 skills 能力

### 5.4 用户故事与功能映射

- 小李最关心：项目理解、代码修改、测试验证、任务恢复
- 小王最关心：多文件执行、生成落盘、非交互调用、skills 复用
- 小黄最关心：边界控制、审批策略、结果可审查、任务可恢复

### 5.5 功能优先级原则

- P0：不具备就无法形成工程闭环的能力
- P1：提升稳定性、可用性和接入效率的能力
- P2：后续扩展能力，如更丰富的集成或生态支持

## 6. 非功能需求

### 6.1 性能

- CLI 启动后应尽快进入可交互状态
- 常规只读任务应在较短时间内输出首轮结果
- Shell 默认超时应可控，避免长时间无响应
- 中小型仓库分析应优先返回结构化结论，而不是长时间沉默

### 6.2 兼容性

- 支持 macOS、Linux、Windows
- 支持主流本地 Shell 环境
- 支持 OpenAI-compatible、OpenAI、Anthropic 类模型接入
- 在本地仓库、远程开发机和 CI Shell 中均可运行

### 6.3 安全性

- 默认只允许在当前工作区内进行文件读写
- 高风险命令必须确认或直接阻断
- 默认不允许越界写入、危险删除和破坏性 Git 命令
- 日志和会话文件应避免泄露敏感信息

### 6.4 可用性

- 命令结构应简单直观，参数体系统一
- 输出应结构化、可读、可复制
- 失败信息应明确说明原因和下一步建议
- 新用户应可通过 `/help` 和诊断能力快速完成环境准备

### 6.5 稳定性与可恢复性

- 单步工具失败不应导致整个会话损坏
- 达到预算上限时必须输出可继续的阶段性总结
- 会话文件损坏时应能跳过并提示告警
- 恢复会话时应优先展示摘要，而不是盲目继续执行

### 6.6 可扩展性

- Provider 层应保持抽象，便于后续替换模型
- 工具层应支持继续扩展新的本地工具能力
- 会话、日志、产物结构应为后续 diff、undo、MCP 等能力预留空间
- 核心 Runtime 不应与某一类 UI 强耦合

## 7. 产品范围

### 7.1 本阶段要做

- 交互式 CLI 会话
- Slash 命令体系
- 非交互式任务执行
- 本地代码读取、搜索、修改和新建
- Shell 命令执行与验证
- 会话持久化、恢复和继续
- 工作区边界与审批控制
- 结果摘要与结构化输出

### 7.2 本阶段不做

- 不做完整 IDE
- 不做多 Agent 协同
- 不做图形化 diff 浏览器
- 不做团队协作与云同步
- 不做远程 skills 市场、插件生态和复杂权限治理后台

## 8. 交互命令设计

### 8.1 Slash 命令建议

```bash
/help
/sessions
/resume <session-id>
/new
/verify
/skills
/skill run <name>
/quit
```

### 8.2 参数原则

- 所有会话命令统一使用 `/` 前缀
- 命令命名尽量短而清晰
- 会话管理、验证、skills 相关命令保持相同语义结构
- 高频命令示例：

```bash
/resume 20260327-001
/skill run go-fix-test
/verify
```

### 8.3 输出原则

- 默认输出对人类可读
- 重要结果有摘要
- 错误信息可复制
- 为自动化场景预留结构化输出能力，例如 JSON 模式

## 9. 技术方案约束

### 9.1 技术栈方向

- 主语言：Go
- 终端交互：优先使用成熟 Go CLI/terminal 方案，并保留非交互命令入口
- 配置管理：轻量且本地优先
- 本地存储：优先文件系统 + JSON / SQLite 级方案

### 9.2 架构原则

1. 单二进制优先
2. 能力分层清晰
3. 本地状态优先
4. Provider 可替换
5. 安全策略集中管理

### 9.3 建议模块划分

- `cmd/`：启动入口与非交互命令
- `internal/config/`：配置加载与校验
- `internal/session/`：会话管理
- `internal/agent/`：任务执行编排
- `internal/cli/`：CLI 会话交互与 Slash 命令
- `internal/tools/`：文件、搜索、Shell 等工具实现
- `internal/skills/`：用户自定义 skills 解析与执行
- `internal/policy/`：安全与审批策略
- `internal/provider/`：模型接入抽象
- `internal/output/`：终端输出与格式化

## 10. 版本规划

### Phase 1：CLI 闭环可用版

- `run` 单次任务模式
- 基础文件工具
- 基础 Shell 执行
- 工作区边界限制
- 基础输出摘要

### Phase 2：CLI 持续会话增强版

- `chat` / 默认交互模式
- `/help`、`/sessions`、`/resume`、`/new`
- 会话持久化
- 更好的错误恢复
- 更清晰的任务状态与工具输出

### Phase 3：用户自定义 skills 与自举验证版

- 用户自定义 skills
- 更细的审批策略
- 更稳定的 Provider 切换
- 在 ByteMind 自身仓库中验证部分功能迭代闭环

## 11. 关键指标与上线标准

- 含文件修改任务的成功率
- 含命令执行任务的成功率
- 会话恢复使用成功率
- 用户自定义 skills 重复调用成功率


## 12. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| CLI 交互负担仍高 | 用户觉得只是“在终端聊天” | 用更清晰的摘要、工具输出和 Slash 命令降低认知负担 |
| 功能范围膨胀 | 研发复杂度失控 | 只做 CLI + 用户自定义 skills，不引入多 Agent、云协作、插件市场等旁支需求 |
| 命令执行风险高 | 用户不敢放权 | 默认收紧边界，审批策略清晰，风险分级明确 |
| 会话恢复不稳定 | 恢复后上下文混乱 | 恢复时优先输出会话摘要和阻塞点 |
| Provider 差异大 | 不同模型行为影响体验 | 在 Provider 层做统一约束，定义最小能力接口 |

## 13. 最终结论

ByteMind 3.0 的正确方向，不是继续停留在“能跑通闭环的 CLI MVP”，也不是扩展成沉重的平台，而是把 CLI 形态做成一个可恢复、可沉淀的 AI Coding Agent。

真正需要优先做好的，不是功能堆砌，而是以下几件事：

1. 更稳定的任务执行闭环
2. 更清晰的 CLI 状态输出与 Slash 交互
3. 更可靠的文件修改与命令执行能力
4. 更清晰的会话恢复与结果总结
5. 能把高频流程沉淀为用户自定义 skills
6. 更严格但不繁琐的安全边界控制

只要这些能力成立，ByteMind 就不是一个“能聊天的终端工具”，而会成为一个开发者愿意在真实仓库里持续使用、甚至可以辅助迭代自身功能的 CLI Coding Agent。
