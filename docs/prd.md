# ByteMind PRD

产品名称：`ByteMind`  
版本：`v0.1`  
产品形态：`本地 AI Coding CLI`

## 1. 产品定位

ByteMind 是一个运行在本地代码仓库中的 AI Coding CLI，目标是帮助开发者通过自然语言完成“理解项目 -> 读取代码 -> 修改文件 -> 执行命令 -> 查看结果 -> 继续迭代”的工程闭环。

## 2. 背景与问题

开发者在使用 AI 辅助编程时，常见问题包括：

- 纯问答式助手只能解释代码，不能直接在仓库内完成操作
- IDE 插件或云端 Agent 往往较重，不适合终端优先的工作流
- 缺少连续会话时，用户每轮都需要重复上下文
- 直接执行命令或修改文件存在风险，需要边界控制与确认机制
- 当模型进入重复调用或执行预算耗尽时，若没有阶段性总结，用户很难接续工作

ByteMind 试图优先解决以上问题，在本地终端场景中提供一个足够轻量、可持续迭代的 Coding Agent 产品。

## 3. 目标用户

- 习惯终端工作流的个人开发者
- 需要在本地仓库中快速定位、修改、验证代码的工程师
- 希望将“读代码、改代码、跑命令”集中在一个 CLI 中完成的技术人员

## 4. 产品目标

### 4.1 核心目标

1. 让用户可以在当前工作区中，通过自然语言完成真实的软件工程操作。
2. 建立可持续的多轮会话机制，使用户可以在同一仓库内连续推进任务。
3. 通过文件工具和 Shell 工具等形成完整执行闭环。
4. 在可用性和安全性之间取得平衡：允许执行、但尽量受控。

### 4.2 成功标准

当用户输入类似以下任务时，产品应能稳定完成：

- “分析这个仓库的结构并给我改造建议。”
- “在当前模块里加一个配置项，并更新读取逻辑。”
- “运行测试，定位失败原因并修复。”
- “继续上一次会话，完成剩余修改。”

闭环成立的标准是：系统能在本地仓库内自主读取上下文、调用工具执行实际操作，并将结果以可继续追问的形式返回给用户。

## 5. 核心场景

### 5.1 仓库理解

- 用户要求分析当前项目结构
- 用户要求定位某个模块、配置入口或业务流程
- 用户要求搜索关键字并解释代码关系

### 5.2 定向代码修改

- 用户要求修改已有函数、配置或模块逻辑
- 系统先搜索并读取文件，再选择精确替换或补丁更新
- 修改后返回变更摘要，必要时继续追问

### 5.3 新增文件或模块

- 用户要求创建新文件、脚本或配置
- 系统根据上下文选择写入文件
- 若涉及多文件，按工具调用逐步完成

### 5.4 命令执行与验证

- 用户要求运行测试、构建或其他命令
- 系统根据审批策略决定是否需要确认
- 根据命令结果继续修复或输出建议

### 5.5 会话延续与恢复

- 用户中断后可恢复历史会话
- 用户可查看最近会话列表
- 用户可在同一工作区新建会话，避免不同任务相互污染

## 6. 核心用户流程

1. 用户准备 `config.json` 设置模型配置。
2. 用户在仓库根目录启动 `bytemind`，或通过 `bytemind run -prompt "..."` 执行单次任务。
3. 系统加载配置、初始化 Provider、创建或恢复会话，并将当前工作区作为执行边界。
4. 用户输入自然语言任务。
5. Agent 基于系统提示词与历史会话，决定是否调用内置工具。
6. 工具返回结果后，Agent 继续决策，直到输出最终答复，或因上下文爆炸/重复调用保护而停止。
7. 每一步消息和工具结果都会被保存到本地会话存储中。
8. 用户可通过 `/sessions`、`/resume`、`/new` 等命令继续管理上下文。

## 7. 初版功能范围

- CLI 聊天模式与单次任务模式
- 本地配置加载
- OpenAI-compatible、OpenAI、Anthropic 三类 Provider 入口
- 工具调用循环
- 文件浏览、读取、搜索、写入、精确替换、补丁修改
- Shell 命令执行与审批策略
- 流式输出
- 本地会话持久化、会话列表和恢复
- 最大迭代预算控制
- 重复工具调用检测与阶段性总结

## 8. 功能需求

| 编号 | 功能 | 优先级 | 需求说明 | 验收标准 |
|---|---|---|---|---|
| FR-01 | 任务输入与运行模式 | P0 | 支持多轮会话模式与单次任务模式，未指定子命令时默认进入聊天模式 | 用户可在仓库根目录启动交互式会话，或通过单次命令执行一个任务 |
| FR-02 | 模型配置与切换 | P0 | 用户可通过配置文件参数切换模型与运行参数 | 用户通过配置文件成功修改配置 |
| FR-03 | 会话管理 | P0 | 支持查看当前会话、列出历史会话、新建会话和恢复会话 | `/session`、`/sessions`、`/new`、`/resume` 可正常使用 |
| FR-04 | 项目上下文获取 | P0 | 系统可浏览目录、读取文件、搜索文本，用于理解当前仓库 | 对“分析项目结构”任务，系统能返回有依据的结果 |
| FR-05 | 文件与代码修改 | P0 | 系统可创建文件、写入文件、精确替换和补丁修改代码 | 用户要求新增或修改文件时，系统能正确完成改动 |
| FR-06 | 命令执行与风险控制 | P0 | 系统可在工作区内执行 Shell 命令，并根据策略直接执行、请求确认或阻止危险命令 | 在 `on-request` 模式下，只读命令可直接执行；修改性或危险命令需确认或被阻止 |
| FR-07 | 执行反馈与结果展示 | P1 | 系统应展示模型输出、工具执行过程和关键结果摘要 | 用户在长任务中能看到工具调用、执行结果和最终结论 |
| FR-08 | 任务保护与可恢复输出 | P1 | 当执行预算耗尽或出现重复工具调用时，系统应停止并给出阶段性总结 | 用户可基于总结继续下一轮任务，而不是丢失上下文 |
| FR-09 | 多 Provider 接入 | P1 | 系统应通过统一协议接入多个 LLM Provider | 同一套消息与工具调用机制可接入 OpenAI-compatible、OpenAI、Anthropic |

## 10. 关键产品原则

- 默认以本地工作区为唯一操作边界，不越界访问其他目录
- 默认先理解上下文，再执行修改
- 默认以结构化工具调用完成任务，而不是让模型只输出建议文本
- 默认将每轮关键状态持久化，保证可恢复性
- 默认对 Shell 执行施加审批和风险控制

## 11. 交互设计要求

### 11.1 输入方式

- 主输入方式为终端自然语言输入
- 支持一次性模式与交互式对话共存
- Slash 命令需要简单、稳定、易记

### 11.2 输出方式

- 回复文本支持流式展示
- 工具调用过程需显示当前执行工具及关键结果
- 最终输出应让用户知道：做了什么、改了哪里、是否完成、下一步如何继续

### 11.3 会话体验

- 会话 ID 需要在启动后可见
- 用户能快速查看最近会话并恢复
- 恢复时需校验工作区一致性，避免跨仓库上下文污染

## 12. 非功能要求

- 运行环境：基于 Go 实现，主应用尽量保持轻依赖
- 跨平台：兼容 Windows 与非 Windows Shell 环境
- 本地性：配置、会话和文件操作均优先在本地完成
- 可观测性：工具执行结果、错误信息和停止原因需要清晰可见
- 稳定性：在模型无法完成任务时，仍应保存状态并输出可恢复结果
- 安全性：危险命令应阻止或要求审批；文件操作需受工作区路径限制

## 13. 成功指标建议

- 用户任务从输入到得到可执行结果的完成率
- 单轮任务中实际调用工具后成功完成的比例
- 会话恢复后继续完成任务的比例
- Shell 审批流程中被拒绝或被阻止的高风险命令数量
- 因预算耗尽或重复调用停止的任务占比
- 用户在最终结果后仍需手动补充大量操作的比例

## 14. 一句话产品定义

ByteMind 是一个面向本地代码仓库的 AI Coding CLI，通过多轮会话、工具调用、受控 Shell 和本地会话持久化，帮助开发者在终端内完成软件工程执行闭环。
